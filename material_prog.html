2.1. Documentation is not an afterthought!

Regardless of the programming language, start with documentation and document your thinking and decisions while you code.

Documentation is not an afterthought!

Step 1 What do we need to know?

  • Describe the problem in our words
  • State the OBJECTIVE of the problem we are trying to solve.
  • Identify the variables for INPUTs and OUTPUTs (ELEMENTS)
  • What constraints need to be identified? (RELATIONS)
  • Model the system: goal + elements + relations. Use simple English (structured) or pseudocode.

Include appropriately in comments. Err on having too many comments. With practice you will learn to make sound and practical choices what constitute too much and what is inadequate amount of comments.

STEP 2 Using HTML5 only, organize your input and output layout. You may want to do separate pages for different states in your application (what does the screen look like to the user when you start the game? what does it look like when you are in the middle of the game? What does it look like when the game finishes?) Fill it with either Lorem Ipsum text and images, or use known representative or sample content, such as integers when you expect the content to be numeric, etc. Once you have the layout completed and code valided, ...

STEP 3 Using CSS3, style all components of your layout. Set up appropriate classes. Minimize the use of ID attributes, but use them if they are used for Javascript purposes. Once you have the layout completed and code valided, ...

STEP 4 Using Javascript, write out in plain English or pseudocode, what functionality you need to develop. Use comments and TODO label to organize your Javascript code. Develop all functionality with the "hard-coded" HTML first. Once everything is working, remove the elements and DOM structures that will be created with Javascript, such as game boards. When you build those structures properly with JS, you should not need to change any CSS or any other Javascript code.

STEP 5 Using PHP (or another server-side programming language), write out in plain English or pseudocode, what functionality you need to develop. Use comments and TODO label to organize your code. Develop all functionality with the "hard-coded" HTML first. Once everything is valid, replace and enhance appropriately with PHP (or another server-side programming langauge).

2.2. Understanding the problem

Even a simple problem needs to be planned, analysed, designed, constructed, and tested before it is implemented. Complex problems can be broken down into a set of simple problems.

Analyze the problem and consider alternate solutions, then select the most appropriate solution for a given situation.
  • Objective: what is the objective of the problem we are trying to solve?
  • Elements: what are the elements of the problem? (output and input)
  • Relations: what are the relations among the elements of the problem? (functionality, processes)
  • Algorithm: what is the step by step process that makes up the problem?
  • Model the system: prototype and test, identify areas of improvement, changes, alternatives, dependencies, constraints...

2.4. Why Progressive Enhancement

Progressive Enhancement

Read & Reflect

Progressive enhancement (PE) is a strategy for web design that emphasizes accessibility, semantic HTML markup, and external stylesheet and scripting technologies. PE consists of a set of core principles:

  • basic content should be accessible to all web browsers
  • basic functionality should be accessible to all web browsers
  • sparse, semantic markup contains all content
  • enhanced layout is provided by externally linked CSS
  • enhanced behavior is provided by unobtrusive, externally linked JavaScript
  • end-user web browser preferences are respected
The PE process consists of three steps:
  • Step One: Build solid foundation with HTML5. Identify content and use the most suitable (meaningful) markup to organize it. All content must be accessible without styles.
  • Step Two: Using well-formed and valid HTML5, style the elements for best presentation. All content must be accessible without scripts.
  • Step Three: Using well-formed and valid HTML5 styled with well-formed and valid CSS3, add behaviour with Javascript.

3.1. Writing Code: standards and conventions

Writing code is similar to academic writing in that when you use or adapt code developed by someone else as part of your project, you must cite your source. However, instead of quoting or paraphrasing a source, you include an inline comment in the code. These comments not only ensure you are giving proper credit, but help with code understanding and debugging.

4.1. Implement Selected Solution

Some notes:

  • Use PE approach to develop web apps.
  • Keep up-to-date with W3 Consortium. For example, we no longer include mime type in style and script elements.
  • Use the same convention as Java and Javascript for code style and documentation (see Tutorial Set 1).
  • Validate HTML5 at validator.w3.org before adding any styles.
  • Validate CSS3 at jigsaw.w3.org before adding any Javascript code.
  1. Organize Your Files
    • Create a project directory (folder) for every web site and every web application you are working on, usually under public_html directory.
    • Each project has its own set of sub-directories. The standard set includes the following sub-directories: css/, images/, media/, content/, scripts/
    • For larger projects, you may wish to use a sub-directory for each language used; for example, instead of creating scripts/ you may create js/, php/ and so on.
    • The project directory contains the home page index.html, the starting page of the web site or web app. All other HTML pages will be stored in the content/ directory.
  2. Focus on the input and output of your web application.
  3. Use "real" content where available.
  4. Use "Lorem ipsum" dummy text and images, as place-holder, where the content is not available ahead of time or during development time.
  5. Do not leave anything blank; do not use repetitious short phrases, or unrelated material as fillers.
  6. Identify and document through comments, what styling and behaviour you intend to implement.
  7. Do not forget to validate all HTML.

9.9. Miscellaneous Resources and Related Tools

P h o t o F i l t r e

PhotoFiltre is a complete image retouching program. It allows you to do simple or advanced adjustments to an image and apply a vast range of filters on it. It is simple and intuitive to use, and has an easy learning curve. The toolbar, giving you access to the standard filters with just a few clicks, gives PhotoFiltre a robust look.

J S C O M P R E S S . C O M

JSCompress.com is an online javascript compressor that allows you to compress and minify your javascript files. Compressed javascript files are ideal for production environments since they typically reduce the size of the file by 30-80%. Most of the filesize reduction is achieved by removing comments and extra whitespace characters that are not needed by web browsers or visitors.

A minifier removes the comments and unnecessary whitespace from a program. Depending on how the program is written, this can reduce the size by about half. An obfuscator also minifies, but it will also make modifications to the program, changing the names of variables, functions, and members, making the program much harder to understand, and further reducing its size in the bargain. Some obfuscators are quite aggressive in their transformations. Some require special annotations in the source program to indicate what changes might or might not be safe. http://yuiblog.com/blog/2006/03/06/minification-v-obfuscation/

C o l o r P i x

ColorPix is a useful little color picker that grabs the pixel under your mouse and transforms it into a number of different color formats. You can use the built-in magnifier to zoom in on your screen, click on a color value to copy it directly to the clipboard, and even keep ColorPix on top of all other apps and out of the way.
