Applying CSS rules to dynamically created elements
Applying Progressive Enhancement principles, all content is marked up in Phase 1, even if you plan or anticipate creating parts of the document with JavaScript at a later point. The goal of this phase is to organize all content, selecting the most appropriate (semantically correct) elements and the output is a complete, well-formed, valid, semantically correct HTML5 code with accessible content. (usable and publishable without CSS and JS)
Start Phase 2 with the output from Phase 1 and style it with CSS (fonts, colours, layouts). Create a framework (a minimal set of CSS rules, properly organized and grouped). The goal of this phase is to style all marked-up content for presentation purpose and the output is a complete, valid CSS3/4 code with accessible content. If done correctly, there should be no changes to the HTML in this phase. (usable and publishable without JS)
Start Phase 3 with the output from Phase 2 and add behaviour with JavaScript (and enhance with server-side functional languages, such as Node.js, PHP, C#, Java, Python, etc.,). Whatever behaviour you add, it should enhance the user-experience, maintain accessibility of the content. When you are replacing a structure, such as a table or a list, you are using the DOM methods to produce exactly the same structure that you created manually in Phase 1. All CSS rules (usually class selectors) that you created and tested in Phase 1, will be applied to the created structure. You can manipulate properties through the CSSOM:
handleToElement.style.backgroundColor = "lightblue";
or
if you already have a class .lightblue in your CSS rule set:
handleToElement.setAttribute("class", "lightblue");

Activity 4. Validation
Form validation is crucial for ensuring that user-submitted data meets the expected criteria. In HTML, you can perform client-side validation to catch errors before the form is submitted to the server. Let’s explore some techniques: Excerpt from MDN
Required Attribute:
The simplest HTML validation feature is the required attribute. To make an input mandatory, add this attribute to the element. When this attribute is set, the element matches the :required UI pseudo-class, and the form won’t submit if the input is empty. An error message is displayed on submission
Built-in Form Validation:
HTML provides built-in form validation features. For instance, if a form field is empty, the required attribute prevents form submission. Example: HTML 

<form action="/submit" method="post">
    <input type="text" name="username" required>
    <input type="password" name="password" required>
    <input type="submit" value="Submit">
</form>
In this example, both the username and password fields are required
Custom Validation:
You can use the pattern attribute to specify a regular expression pattern that the input value must match. Example: HTML 

<input type="text" name="phone" pattern="\d{3}-\d{4}" placeholder="xxx-xxxx">
This ensures that the phone number is entered in the format xxx-xxxx
JavaScript Validation:
While HTML validation is useful, you can enhance it with JavaScript. Use event listeners (e.g., blur, change) to check field validity sooner (before form submission). Example: JavaScript 

const inputElement = document.querySelector('input[name="email"]');
inputElement.addEventListener('blur', () => {
    if (!inputElement.checkValidity()) {
    // Display an error message or style the input
    }
});
Here, we check the validity of the email input when the user moves away from it (on blur)
Client-side validation improves user experience, but it’s not exhaustive for security. Always perform server-side validation to prevent malicious data from reaching your server.
